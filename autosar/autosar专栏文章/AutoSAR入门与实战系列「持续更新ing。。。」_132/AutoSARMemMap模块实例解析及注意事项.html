1 AUTOSAR Memory Mapping的运行机制

AUTOSAR Memory Mapping用来干啥的？

  • 它是在抽象编译器之后 （abstracting the compiler）的一种在内存中定位代码和变量（code and variables）的机制。
  • 内存映射通过将依赖于编译器的代码集中在特定的头文件中来抽象编译器。

1.1 AUTOSAR Memory Mapping实例解析

让我们看一下AUTOSAR Memory Mapping规范中描述的示例【AUTOSAR_SWS_MemoryMapping.pdf】。

代码1：

//BSW Module
  1 #define EEP_START_SEC_VAR_INIT_16
  2 #include "MemMap.h"
  3 static uint16 EepTimer = 100；
  4 static uint16 EepRemainingBytes = 16；
  5 #define EEP_STOP_SEC_VAR_INIT_16
  6 #include "MemMap.h"

在上面的示例中，名为 Eep（EEPROM Driver）的模块定义了全局变量 EepTimer（第 3 行）和 EepRemainingBytes（第 4 行）。在这个变量定义前后，包含了宏定义EEP_START_SEC_VAR_INIT_16（第1行）/EEP_STOP_SEC_VAR_INIT_16（第5行）和头文件“MemMap.h”（第2、6行）。

MemMap.h 的示例在AUTOSAR Memory Mapping中描述如下。
代码2：

//MemMap.h 
   1 #ifdef EEP_START_SEC_VAR_INIT_16 
   2 #undef EEP_START_SEC_VAR_INIT_16 
   3 #define START_SECTION_DATA_INIT_16 
   4 #elif 
   5 /* 
   6 additional mappings of modules sections into project 
   7 sections 
   8 */ 
   9 ... 
  10 #endif 
  11 
  12 
  13 #ifdef START_SECTION_DATA_INIT_16 
  14 #pma section data "sect_data16" 
  15 #undef START_SECTION_DATA_INIT_16 
  16 #undef MEMMAP_ERROR 
  17 #elif 
  18 /* 
  19 additional statements for switching the project sections
  20 */ 
  21... 
  22 #endif

如果将代码2（MemMap.h）替换到代码1种，编译器（预处理器）将做处理，您将看到在变量（EepTimer（第 3 行）和 EepRemainingBytes（第 4 行））之前定义的宏 EEP_START_SEC_VAR_INIT_16 在 MemMap.h 中启用以下编译指示：

  14 #pragma section data "sect_data16"

1.2 编译器的选择

根据编译器的不同，pragma 指定段的方法也有所不同，但在上述示例中，在“section data”之后指定要放置目标变量的段的名称（sect_data16）。通过这种机制，可以在不依赖编译器的情况下开发源代码，如上例中的BSW Module。

通过pragma方法可能不适用于所有编译器。Memory Mapping 规范只考虑了 GreenHills 和 TASKING 等 5 个特定的编译器，并表示任何不支持上述机制的编译器都不受 AUTOSAR 支持。

2 内存分配关键字

EEP_START_SEC_VAR_INIT_16/EEP_STOP_SEC_VAR_INIT_16 等宏称为“内存分配关键字”，为每个目标部分【target section 】（代码、变量、CONST 变量等【code, variables, CONST variables,】）指定语法。下面是变量的内存分配关键字语法示例。

<PREFIX>_START_SEC_VAR_<INIT_POLICY>_<ALIGNMENT>
<PREFIX>_STOP_SEC_VAR_<INIT_POLICY>_<ALIGNMENT>

  • 上面的例子，是“INIT”，表示它是一个有初始值的变量（variables）。还有其他选项，例如 NO_INIT（不初始化）和 CLEARED（零初始化），并且根据初始化策略，您可以选择将其放置在哪个部分（section），例如 .bss 或 .data。

  • 数字16表示放置变量的对齐方式。在上面的例子中，它是“16”，但是由于EepTimer和EepRemainingBytes是16位无符号整数，所以指定16位对齐。将具有相同对齐方式的变量分组到相同的部分允许一些编译器实现最佳内存放置。

  • 另外，Memory Allocation Keyword的语法也有其他详细规定，有兴趣的可以参考Memory Mapping规范。

3 如何生成BSW和 SWC的MemMap.h文件

  • MemMap.h 是为 BSW 准备的一个通用文件。对于 BSW，它包含了所有 BSW 内存分配关键字的定义。-从 R4.1 开始，每个 定义BSW 内存分配关键字的文件都是分开的。
  • 对于 SWC， SWC内存分配关键字的定义包含在如“xxx_MemMap.h”的各个文件中。
  • 手动创建所有 MemMap.h并不容易，Memory Mapping 还规定了生成器根据每个 BSW 和 SWC定义的信息以及为每个编译器 设置的信息来生成 MemMap.h 的规范。 两者均已标准化为 arxml 格式和 AUTOSAR 规范。
    • 在为每个BSW和SWC定义的信息中，设置了称为MemorySection的信息。为该组件的实现中出现的每个内存分配关键字设置一个 MemorySection。
    • 每个 MemorySection 还包含对 SwAddrMethod 的引用,SwAddrMethod 就像是 MemorySection 的公共组。
    • 对于为每个编译器设置的信息，根据所使用的 SWC、BSW MemorySection 和 SwAddrMethod, 通过设置pragma来实现分配section和section的对齐方式。这里pragma设置的粒度可以是MemorySection （MemMapSectionSpecificMapping）或SwAddrMethod（MemMapGenericMapping）。

例如，假设您有 MemorySections
“VAR_NO_INIT_8”、“VAR_NO_INIT_16”、“VAR_NO_INIT_32”，并且所有 MemorySections 都引用“VAR_NO_INIT”的 SwAddrMethod。如果要将属于三个 MemorySection 的变量放在不同的部分中，请使用 MemMapSectionSpecificMapping 并设置每个 pragma。

如果您可以将它们全部放在同一部分，则使用 MemMapGenericMapping 允许您仅设置一个 pragma。

这样，在实际进行内存分配时，可以根据所使用的编译器进行适当的内存分配。

4 编译优化的影响

即使您可以使用一个编译器按预期分配该部分，但使用不同的编译器也可能会出现意外现象。例如，当指定代码大小优先编译选项时，可以从函数中链接和调用特定于编译器的共享函数以减小大小。

在这种情况下，分配共享函数的段固定为standard section，它可能与调用函数分配的的段【section】不同。

近年来的编译器有各种优化功能，除此之外，根据编译器的不同，可能会出现意外的部分，因此建议在链接后检查最终section信息。记住，检查最终的map文件很重要。
