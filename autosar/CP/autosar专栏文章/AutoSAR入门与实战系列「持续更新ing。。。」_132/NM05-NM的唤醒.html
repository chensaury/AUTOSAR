Nm 不负责节点或网络的正常唤醒，这将由 COM 管理器 ( ComM ) 完成。

1 唤醒

  1. 同步唤醒的流程如下： Nm通过调用 ComM_Nm_NetworkStartIndication() 将来自网络的唤醒指示（BusNm 调用Nm_NetworkStartIndication()）转发给ComM，然后ComM将调用Nm_PassiveStartUp，通过 Nm 转发到BusNM 的相应接口。

  2. If the configuration setting ‘Coordinator Request Channels In Bus Sleep’ is not ON ，ComM 可以配置为在请求一个网络时自动启动所有网络（同步唤醒）。

  3. If the configuration setting ‘Coordinator Request Channels In Bus Sleep’ is ON ，如果协调器集群内的一个网络被总线唤醒或由 ComM 请求，而 此时NM 协调器处于“关闭”状态，则所有网络在协调器集群中醒来。因此，此功能允许 NM 协调器仅唤醒一个 NM 协调器集群中的通道，而不是像 ComM 的“同步唤醒”那样的每个通信通道。

  4. 如果NM 协调功能被激活并且协调关闭已经在NM 协调集群上启动，取决于协调器算法配置，在每个包含的总线实际释放之前可能需要一些时间。如果协调总线之一上的任何节点更改其状态并在所有网络释放之前开始请求网络，则协调算法中可能会Abort Shutdown。如下图所示，这可以通过四种方式发生：

    1）网络上尚未释放且仍处于“网络模式”的节点再次开始请求网络。这将由总线特定的 NM检测到，该 NM 将通过调用Nm_RemoteSleepCancellation()通知Nm。

    2）网络上已被释放并指示“准备总线睡眠模式”但未指示“总线睡眠模式”的节点再次开始请求网络。这将由总线特定的 NM检测到，该 NM将自动将状态更改为“网络模式”并通过调用Nm_NetworkMode()通知Nm。

    3）ComM在NM 协调集群中的任何网络上请求网络并通过调用Nm_NetworkRequest()or Nm_PassiveStartUp()通知Nm。

    4）主动协调该网络的协调器发送 Nm 消息，并清除 Ready-Sleep Bit。这将被BusNM （总线特定NM）检测到（仅在被动协调的通道上）并通过调用Nm_CoordReadyToSleepCancellation()转发到 NM 。

注意：一旦出现上面的四种唤醒方式，通用方法是中止关闭并再次开始请求网络。但是，已经进入“总线睡眠模式”的网络不会被自动唤醒。
