1 CommonAPI的介绍

CommonAPI 是由 GENIVI 开发的 C++ 框架，用于遵循 SOA 的进程间和网络通信。其目标是为不同的通信框架或协议提供统一的应用程序接口 (API)。使用 CommonAPI 的项目的基础是存在由开发人员共享的服务定义。为此，CommonAPI 使用了 Franca 接口定义语言 (Franca IDL)。这种语言允许使用方法、事件和字段定义服务，以及项目特定数据类型的规范，例如结构或枚举，然后将具有特定的在线表示。这类似于 SOA 中存在的消息格式规范，因为它包含服务或客户端对消息进行编码和解密所需的所有信息。从这个 Franca 文件中，core-tools ，为服务提供者（称为Skeletons ）和消费者（称为Proxies ）生成代码。前者包含指定方法的默认实现，而后者具有在内部处理对服务提供者的调用和整个网络过程的方法。然后，开发人员将通过 C++ 继承扩展生成的代码，而无需实际修改生成的代码文件。这个生成的代码使用来自 CommonAPI Runtime 的方法，这是来自底层通信协议的实际抽象层。基本运行时称为核心运行时并且具有 SOA 的高级概念，例如远程方法、事件和字段。这两个核心项目都代表了程序员在使用此框架进行开发时必须与之交互的 API。图1展示了这种交互。然后为了指定一个实际的中间件，需要在编译过程中采取额外的步骤。

图 1：CommonAPI 功能模块和相关抽象级别概述

迄今为止支持的通信协议是 SOME/IP、D-Bus（一种用于同一系统内的进程间通信的协议）以及仍在开发中的 Web 应用程序消息传递协议 (WAMP) 。对于这些协议中的每一个，都有一个针对该协议的各自核心项目的绑定，例如，使用 SOME/IP 的代码生成器称为someip-tools，并且运行时称为someip-runtime ，相同的逻辑适用于 D-Bus 和 WAMP。为了使用这些协议之一，需要生成相应协议的代码以及在编译时将相应运行时链接到应用程序。每个协议运行时都包含宏，用于在运行时动态链接各自的绑定到核心运行时，有效地为其提供网络容量。通过这个过程，开发人员可以抽象出不同通信的内部工作原理协议，并且只能专注于他们提供的熟悉的服务。开发人员关注的这种转变也使得采用新技术变得非常容易。使用 CommonAPI 开发项目后，更改为另一种技术只需重新编译，甚至无需更改项目。此外，由于这遵循 SOA，并且每个 Franca 文件都会生成消息传递方案和提供的服务，因此并发开发很简单，其中一个团队负责服务提供者系统，另一个团队负责服务客户端. 由于生成的代码对于同一个 Franca 文件是相同的，因此重点从项目的网络部分转移到其实际功能需求。

2 AUTOSAR AP通讯管理介绍及改进

与 CommonAPI 类似，ara::com 的目的是将开发人员从机制中抽象出来，以查找和连接应用程序。它还遵循 SOA，允许在同一设备或不同设备中的应用程序之间提供和使用服务。这些服务由方法、事件和字段组成，其发现可以通过使用服务发现静态或动态建立。相应地，基于服务规范文件的代码生成也被标准化。基于 AUTOSAR 元模型，为服务提供者骨架和服务消费者代理生成代码然后可以由开发人员扩展。实际的通信协议是抽象的。应用程序只与 ara::com API 交互，然后由它处理与实际协议的交互。这种方法类似于 GENIVI CommonAPI 方法，但 AUTOSAR 还需要 CommonAPI 或任何其他现有平台未提供的附加功能。**即，接收端缓存，已经存在于 CP 中，具有零复制能力的 API，可以将内存管理转移到中间件和数据接收过滤 。因此AUTOSAR在现有的基础上设计了一个通信中间件API，以满足其需求。

ara::com 所基于的主要协议是 SOME/IP。由于它已经设计用于汽车用途，因此符合 AUTOSAR 要求。它成为 ara::com 支持的第一个协议，这是 AUTOSAR 要求，任何开发的 AUTOSAR AP 兼容解决方案都必须支持 SOME/IP，尽管底层中间件的实际选择取决于应用程序开发人员。另一个研究的协议是数据分发服务（DDS），这是一个对象管理组中间件标准，它建立在所有感兴趣的应用程序都可以访问的“全局数据空间”的概念之上。DDS 已应用于不同的安全关键系统，例如空中交通管制、运输和医疗系统 。在 ara::com 创建时，DDS 严格遵循发布-订阅通信机制，不支持请求-响应。因此，它不符合成为 ara::com 可能的中间件的一部分所需的标准。ara::com 发布后不久，DDS 在其模型 中引入了 RPC 的可能性，几个月后 AUTOSAR 将 DDS 绑定添加到 ara::com 将 DDS 添加到应用程序通信的可能中间件中。

最后，AP 最终得到了一个类似于 GENIVI 的工作流程。因此，当使用相同的通信协议时，AUTOSAR 和 GENIVI 或其他平台之间的互操作性自然而然。平台上下文切换在中间件之上一层完成，在 AUTOSAR AP 情况下使用 ara::com，在 GENIVI 情况下使用 CommonAPI。
