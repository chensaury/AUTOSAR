1 SOA面向服务的通信交互


如上图所示，女神去热水澡堂洗澡，想搓背（find服务），于是她付要付搓背钱给澡堂老板，这时澡堂老板知道通过小王和小明的毛遂自荐（offer服务），知道现在澡堂有小王小明都会搓背，由于此时小明正在忙着搓背呢，而小王闲着呢。于是澡堂老板安排小王给女神搓背，给了她一个搓背牌子，女神泡完热水澡，拿着搓背牌子找到小王给她搓背。
注意图中有三个角色：

  • 中间件：实现服务的注册/订阅/发布等功能
  • 服务提供方（Service Provider）：实现服务功能（包括方法，事件和字段）的服务端
  • 服务消费者（Service Consumer）：使用服务实例的客户端
    有两种服务机制：
  • Offer Service
  • Find Service

2 SOME/IP-SD的主要功能

我们在「SOME/IP概述2【SOME/IP的主要中间件功能+SOME/IP报文PDU的封装】」概述了SOME/IP 的主要中间件功能。其中服务发现 (SD) 协议是 SOME/IP 的核心功能。在面向服务的架构中，服务（即函数功能实体【方法、事件或字段】）必须是可发现的。
SOME/IP SD有以下几个主要任务：

  • （1）SOME/IP SD 协议管理提供动态查找功能实体（即服务）并能配置其访问权限的功能【是提供服务还是阻止它可用】；
  • （2）除此之外SOME/IP SD 协议管理向网络发送事件消息的需求（发布/订阅）。这允许仅向需要它们的接收者发送事件消息。
  • （3）检测服务实例是否正在运行。

3 动态服务发现的机制

为了让客户端知道哪些服务当前可用，SOME/IP-SD 提供了两种允许动态发现服务的机制。借助动态服务发现的机制，不同的 ECU 可以提供服务实例并在车辆网络中找到可用的服务实例「服务实例是由服务接口定义的服务的单一实现」。

3.1 Find Service

注意：下文中的条目可以理解成消息。

如上图中的find搓背服务，查找服务允许客户端请求可用的服务，也就是通过发现服务在网络中来识别并找到可用服务实例及其位置的操作。SOME/IP的某个服务有唯一的服务编号，如洗澡服务的ID是0x1234。
注意：Find Service 类型报文用于查找服务实例，但是只在服务的当前状态未知时才发送。（未收到当前Service Offer并且Service仍然有效）。
Find Service有以下几点要求：

1 、查找服务条目应按以下方式设置条目字段：

  • 类型应设置为 0x00（查找服务）。

  • 服务 ID 应设置为应找到的服务的服务 ID。如果应返回所有服务实例，则实例 ID 应设置为 0xFFFF。如果只返回单个服务实例，则应设置为特定服务实例的Instance ID。

  • Major Version 应设置为 0xFF，即返回任何版本的服务。如果设置为不同于 0xFF 的值，则应仅返回具有此特定主版本的服务。

  • Minor Version 应设置为 0xFFFF FFFF，这意味着应返回任何版本的服务。如果设置为不同于 0xFFFF FFFF 的值，则应仅返回具有此特定Minor Version的服务。

  • TTL 应设置为 Find Service 条目的生命周期。在此生命周期之后，Find Service 条目将被视为不存在。也就是在一定时间后会停止寻找这个服务。
    如果 TTL 设置为 0xFFFFFF，则 Find Service 条目应始终被视为有效，直到下一次重新启动为止；TTL 不应设置为 0x000000，因为这被认为是停止提供服务入口

3.2 Offer Service

提供服务允许服务器向所在的网络中的其他ECU声明其所能够提供的服务（服务是通过ServieID标识的）。Offer Service有以下几点要求：
1 、Offer Service 条目应按以下方式设置条目字段：

  • 类型应设置为 0x01（OfferService）。

  • 服务 ID 应设置为提供的服务实例的服务 ID。

  • Instance ID 应设置为提供的服务实例的 Instance ID。

  • 主要版本应设置为提供的服务实例的主要版本。

  • 次要版本应设置为提供的服务实例的次要版本。

  • TTL 应设置为服务实例的生命周期。在此生命周期之后，服务实例将被视为未提供。如果 TTL 设置为 0xFFFFFF，则提供服务条目应被视为始终有效，直到下一次重新启动。TTL 不应设置为 0x000000，因为这被认为是停止提供服务入口。

2 、提供服务条目应始终至少引用一个 IPv4 或 IPv6 端点选项，以表明服务的地址和端口，告诉客户端通过这个地址来访问。
3 、对于服务所需的每个传输层协议（即 UDP 和/或 TCP），如果支持 IPv4，则应添加 IPv4 端点选项（如下图所示）；如果支持 IPv6，则应添加 IPv6 端点选项。

  • SOME/IP 应允许服务同时使用 UDP 和 TCP。消息由哪个底层传输协议发送，应由配置确定：服务可以同时使用 UDP 和 TCP 端点。但是每个服务元素（即方法和事件和Filed）都应该配置是使用 TCP 还是 UDP；也就意味着不能通过 TCP 和 UDP 提供相同的事件。
  • 在 UDP 的情况下，端点选项指的是事件和通知事件的源地址和源端口，它也是客户端可以向其发送方法请求的地址；在 TCP 的情况下，端点选项指的是 IP 地址和端口，客户端需要打开 TCP 连接才能使用 TCP 接收事件。

4 发布订阅的机制

通过发布订阅的机制，客户端可以只接收自己感兴趣的事件。通常提供服务的服务器会将事件进行分类成为Event Group ，客户端可以通Subscribe 事件消息实现订阅Event Group ID。

4.1 订阅事件组

1 、订阅事件组条目应按以下方式设置事件组条目字段：

  • 类型应设置为 0x06（订阅事件组）。

  • 服务 ID 应设置为包含订阅的事件组的服务实例的服务 ID。

  • Instance ID 应设置为包含订阅的事件组的服务实例的 Instance ID。

  • 主版本应设置为订阅的事件组的服务实例的主版本。

  • 事件组 ID 应设置为订阅的事件组的事件组 ID。

  • 次要版本应设置为订阅的事件组的服务实例的次要版本。

  • TTL 应设置为订阅的生命周期。如果设置为 0xFFFFFF，则订阅事件组条目应被视为有效，直到下一次重新启动。TTL 不应设置为 0x000000，因为这被认为是停止提供服务入口。

  • Reserved应设置为 0x00，直至另行通知。

  • 如果客户端按顺序发送第一个订阅以触发初始事件的发送，则初始数据请求标志应设置为 1。否则设置为 0。

  • Reserved2 应设置为三个 0 位。

  • 计数器应用于区分并行订阅同一服务的同一事件组（仅端点Endpoint不同）。如果不使用，设置为 0x0。订阅事件组条目应引用一个或两个 IPv4/IPv6 端点选项（一个用于 UDP，一个用于 TCP）

4.2 订阅事件组确认

1 、订阅事件组确认条目类型应用于指示订阅事件组条目已被接受。
2 、订阅事件组确认条目应按以下方式设置条目字段：

  • 类型应设置为 0x07 (SubscribeEventgroupAck)。
  • 服务 ID、实例 ID、主要版本、事件组 ID、TTL、保留、初始数据请求标志、保留 2 和计数器应与请求的订阅事件组中的值相同。
  • 通过多播传输的事件和通知事件的订阅事件组确认条目应引用 IPv4 多播选项和/或 IPv6 多播选项。多播选项表明了事件和通知事件将被发送到的多播地址和端口。
