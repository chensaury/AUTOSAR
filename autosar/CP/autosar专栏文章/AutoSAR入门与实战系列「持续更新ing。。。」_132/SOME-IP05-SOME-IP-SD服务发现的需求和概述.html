1 SOME/IP-SD所处协议栈中的位置

如下图所示，SOME/IP-SD 依赖于 SOME/IP。SOME/IP 本身支持 TCP 和 UDP 通信，但 SOME/IP SD 限制只能通过 UDP 使用 SOME/IP

2 SOME/IP-SD 用到的缩略语

缩写/首字母缩略词：描述：
Method方法、过程、函数或被调用的子例程。
Parameters方法或事件的输入、输出或输入/输出参数
Remote Procedure Call (RPC)使用消息来从一个 ECU 到另一个 ECU 的方法调用
Request客户端向服务器调用方法的消息
Response服务器向客户端传输方法调用结果的消息
Request/Response communica-tion由请求和响应组成的 RPC
Fire&Forget communication仅包含请求消息的 RPC 调用
Event一种单向数据传输，仅在changes时调用或循环调用，并从数据的producer生产者发送到consumers消费者。一个事件甚至可以周期性地和在变化时自发地发送。这完全是发送应用程序的责任，因为接收者没有办法区分这两者。
Field字段表示状态，有三种方法：getter、setter 和 notfier，见下文描述
Notification Event字段通知器（the notifier of an field） 发送的事件消息。此类通知者的消息无法与事件消息区分开来；因此，当提到（ refering to ）一个事件的消息时，这应该也适用于字段通知者的消息。
Getter允许对字段Field进行读取访问的请求/响应调用。
Setter允许对字段Field进行写访问的请求/响应调用。
Notifier在消息字段Field的值更改时发送具有新值的事件消息。
Service零个或多个方法methods、零个或多个事件events以及零个或多个字段fields的逻辑组合（允许空服务，例如用于在 SOME/IP-SD 中声明非 SOME/IP 服务）。说人话就是一个离散功能单元，我们可以封装成一个函数来实现这个功能
Eventgroup服务中的事件或字段的通知事件的逻辑分组「分组为了支持订阅subscription」
Service Interface服务「包括其方法，事件和字段」的正式规范（formal specification ），说人话就是能够被其他模块调用的函数名称/API ，服务通过这个函数名称/API被其他ECU所使用
Service Instance服务接口的软件实现，可以在车辆上或ECU 上存在不止一次 ，说人话就是一个函数名称/API的定义和实现
Server提供服务实例的 ECU
Client使用服务器服务实例的 ECU
Union or Variant一种动态假设不同数据类型的数据结构。
Offering a service instance一个 ECU 实现一个服务实例并告诉其他使用SOME/IP-SD 的ECU ，他们可能使用这个服务实例
Finding a service instance发送 SOME/IP-SD 消息以找到所需的服务实例。
Requiring a service instance向实现所需服务实例的 ECU 发送 SOME/IP-SD 消息，这意味着其他 ECU 需要该服务实例。如果服务实例未运行（即尚未提供 ），也需要发送 SOME/IP-SD 消息。
Releasing a service instance向托管/实现此服务实例的 ECU 发送 SOME/IP-SD 消息，这意味着该服务实例不再需要。
Server-Service-Instance-Entry本地 ECU 提供的服务实例的配置和所需数据（在提供此服务的ECU（服务器）上）
Client-Service-Instance-Entry另一个 ECU 提供的服务实例的配置和所需数据（在使用此服务（客户端）上）
Publishing an eventgroup使用一条 SOME/IP-SD 消息，向其他 ECU 提供服务实例的事件组（eventgroup）
Subscribing an eventgroup使用 SOME/IP-SD 消息需要服务实例的事件组（eventgroup）

3 SOME/IP-SD 协议的需求

  • SOME/IP 服务发现协议应提供运行时发现服务和通信路径的功能。「服务发现与基于服务的通信相结合，允许在系统设计阶段未预定义的通信，即通信伙伴之间的通信不是在系统设计期间静态定义的」

  • SOME/IP 服务发现协议应在 SOME/IP 协议之上使用序列化和通信服务发现相关的消息「SOME/IP 服务发现专门用于基于 SOME/IP 的通信，通用功能只需设计和实现一次」，如下图所示

  • SOME/IP 服务发现协议应支持单播服务发现「从一个发送者只发送给一个接收者的消息」，如提供服务的 SD 消息作为查找消息的确认发送到特定的一个接收器

  • SOME/IP 服务发现协议应支持多播服务发现「从一个发送者发送到多个接收者的消息」，如提供服务的 SD 消息通过 IP 多播发送。

  • SOME/IP 服务发现协议应支持发布和订阅属于 SOME/IP 和非 SOME/IP 服务的数据。注意：可以使用 SOME/IP SD 发现和订阅非 SOME/IP 服务则可以实现非 SOME/IP 服务可以在没有 SOME/IP 的情况下使用。

  • SOME/IP 服务发现协议应支持不同版本的具有不同功能级别的相同服务。

  • SOME/IP 服务发现协议应定义服务发现报文标头和有效负载的格式。

  • SOME/IP 服务发现协议应定义有序的功能集，以符合提供不同级别功能的实现，但每个实现都应支持基本功能「最小功能集」，这可以保证使资源较少的小型 ECU 可以支持服务发现。

  • SOME/IP 服务发现应以分布式方式运行，无需以集中管理方式运行

  • SOME/IP 服务发现应支持可配置的时间以确保经过一定时间后发生状态变化。这为了满足不同的车载网络时间配置要求。

  • SOME/IP 服务发现协议应支持查找本地网络中的服务实例「因为要与另一个位置未知的服务实例进行通信」

  • SOME/IP 服务发现消息应包含如何联系通讯伙伴的信息。该如何与通信伙伴联系的信息应了解底层传输协议。「因为与远程通讯伙伴端点建立通信的信息取决于使用的传输协议，比如底层传输协议是支持 IPv4 还是 IPv6」。

  • 除了传输标准化数据，SOME/IP 服务发现协议还应支持传输可选数据。「因为在服务发现期间，可能需要额外的非标准化数据，比如基于附加条件的实例选择」

  • SOME/IP 服务发现协议应提供负载平衡支持来实现平均分配服务订阅。「比如多个提供商向订阅者提供相同的服务。负载应均匀平衡。为此要在系统设计阶段进行负载平衡」

  • SOME/IP 服务发现协议应支持检测已发布服务实例是处于活动状态还是终止状态并通知订阅者。「因为服务可以不正常地关闭；一些应用程序想要检测到这一点。」

  • SOME/IP 服务发现协议应支持主动向本地网络内的其他通信伙伴提供已发布的服务。「可以用在没有循环查找消息的情况下检测可用服务」

  • SOME/IP服务发现协议应支持主动停止提供服务来通知其他通讯伙伴。「因为当一个发布的服务被正常终止时，有必要通知其他通讯伙伴该服务不再可用」

  • SOME/IP 服务发现协议应支持订阅由服务实例提供的事件。「因为基于服务的事件通信基于发布/订阅模式来控制通讯，所以服务提供方应支持接收事件订阅机制」

  • SOME/IP 服务发现协议应支持主动拒绝请求的订阅事件。「如果发布者无法将请求的事件提供给请求者，则有必要拒绝并通知订户，比如在发布者超载的情况下」

  • SOME/IP 服务发现协议应支持主动停止一个事件「已订阅的」的订阅。「当接收方不再需要事件的数据时「即不再使用另一方提供的事件」，需要停止通讯。」

  • SOME/IP 服务发现协议应支持检测服务提供方是否已重新启动。「ECU 重启可能导致服务提供方的内部状态丢失，从而导致不符合订阅者期望的数据。」

  • 如果发生意外情况， SOME/IP 服务发现协议应标准化错误处理。「如果发生意外情况，需要进行优雅的错误处理。协议需要支持这一点，以使应用程序有可能对这些做出反应。」

  • SOME/IP 服务发现协议应支持服务发现功能「例如服务提供，可能超期」的 TTL机制。「TTL 防止服务发现状态（例如，提供服务）持续存在，因为由于错误的实现、配置或系统中的其他错误而忽略了停止提供服务的状态更新。这种机制有利于将服务发现的错误状态限制在最长的时间内。」
